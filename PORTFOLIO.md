# Limited Market — 한정판 상품 선착순 구매 API

> 짧은 시간에 트래픽이 몰리는 **한정판 선착순 판매** 상황에서,
> **재고가 단 1개도 초과 판매되지 않는 것**을 목표로 동시성 제어와 데이터 정합성에 가장 공을 들인 프로젝트입니다.

[GitHub ↗](https://github.com/HaheeBahee) · [API Docs (Swagger) ↗](#)

---

## 1. 프로젝트 목표

이커머스에서 한정판 상품 판매가 오픈되는 순간, 수천 명이 동시에 구매 버튼을 누릅니다.
이때 재고보다 많은 주문이 접수되거나, 재고가 음수로 내려가거나, 취소 후 재고가 제대로 복원되지 않는 문제가 발생할 수 있습니다.

이런 문제들이 **코드의 어느 지점에서, 왜 발생하는지 직접 재현해보고 싶었습니다.**
그리고 해결 방법으로 알려진 여러 방식(낙관적 락, 비관적 락, Redis 원자 연산)을 모두 구현해
**각 방식의 트레이드오프를 테스트로 비교한 뒤, 이 상황에 맞는 구조를 근거를 갖고 선택하는 것**이 이 프로젝트의 목표였습니다.

최종적으로 **데이터 정합성을 깨지 않으면서 높은 처리량을 유지하는 주문 시스템**을 설계·검증했습니다.

## 2. 시스템 개요

한정 수량 상품을 등급(VIP / 일반)별로 오픈 시간을 다르게 두고 선착순 판매하는 커머스 백엔드 API입니다.

- 회원 등급별 구매 오픈 시간 차등 (VIP 선오픈 → 일반 오픈 → 마감, 스케줄러 기반 상태 전환)
- 주문 생성 → 결제 → 배송 생성 → 취소/환불까지의 전체 주문 라이프사이클
- 주문 상태 변경 이력(OrderStatusHistory) 추적
- JWT 기반 인증 + Redis 기반 Refresh Token 관리

```
Client → Spring Boot API ─┬─ MySQL  (주문/결제/배송 — 영속 데이터, 비관적 락)
                          └─ Redis  (재고 카운터, Refresh Token — 빠른 선착순 필터)
```

## 4. 핵심 문제 해결 (AS-IS → TO-BE)

### 4-1. 선착순 재고 차감 동시성 제어 — 락 전략의 진화 ⭐ 가장 공들인 부분

**AS-IS — 문제를 실측으로 증명**

단순한 `재고 조회 → 차감 → 저장` 로직에 JMeter로 **동시 1,000명 주문**(재고 100개)을 보내자:

> 주문 성공 **208건** (정상이라면 100건) / 남은 재고 **5개** (정상이라면 0개)

두 트랜잭션이 같은 재고를 읽고 각각 차감하는 **Lost Update**로 정합성이 완전히 깨졌습니다.

**과정 — 세 가지 전략을 구현하고 JMeter로 실측 비교**

| 전략 | 실측 결과 | 판단 |
|---|---|---|
| ① 낙관적 락 (`@Version` + `@Retryable`) | 충돌 환경(재고 5개)에서 에러율 50~90%, **재고가 남았는데도 재시도 3회 소진으로 주문 실패** + 데드락 발생 | 충돌이 기본인 선착순에 부적합 |
| ② 비관적 락 (`SELECT ... FOR UPDATE`) | 동일 조건에서 초과 판매 0건, 저경합 구간 에러율 0% | 정합성 확보, 단 모든 요청이 DB 락 경유 |
| ③ ② + Redis DECR 선차감 (최종) | 품절 요청을 DB 도달 전 차단 → 고부하에서 TPS·응답시간 모두 개선 | 채택 (정합성 + 처리량) |

**트러블슈팅 — 낙관적 락에서 발견한 데드락의 원인 (MySQL gap lock)**

낙관적 락 테스트 중 데드락이 발생해 원인을 추적했습니다. 주문 트랜잭션은 `order_item INSERT`와 `sale UPDATE`를 함께 수행하는데:

- `INSERT`가 잡는 **gap lock은 여러 트랜잭션이 동시에 보유 가능**하지만,
- `UPDATE`에 필요한 **row lock은 상대 트랜잭션의 gap lock이 풀려야 획득 가능** →
  두 트랜잭션이 서로의 gap lock 해제를 기다리며 **데드락**

비관적 락은 트랜잭션 시작 시점에 row lock을 먼저 잡아 진입 자체를 직렬화하므로 이 구조에서 데드락이 발생하지 않습니다.
즉 "코드 버그"가 아니라 **INSERT + UPDATE가 공존하는 주문 도메인의 구조적 특성**이 락 전략 선택의 근거였습니다.
추가로 여러 상품 주문 시 **saleId 오름차순 정렬 후 락 획득**으로 락 순서 불일치 데드락도 차단했습니다.

**TO-BE — Redis DECR 1차 필터 + DB 비관적 락 최종 방어선**

1. 주문 요청 시 Redis 원자적 `DECR`로 선차감 — 음수면 즉시 `INCR` 복구 후 품절 응답 (DB 부하 0)
2. Redis를 통과한 요청만 DB 트랜잭션 진입, `FOR UPDATE`로 Sale 행을 잠그고 실제 차감
3. DB 트랜잭션 실패 시 Redis 재고 즉시 복구
4. `@Version`은 비관적 락과 역할이 중복되어 **의도적으로 제거** (불필요한 버전 충돌 예외 방지)

**검증 — JMeter 부하 테스트 (재고 100개, 초과 판매 0건 유지)**

| 동시 요청 | 비관적 락 단독 TPS | + Redis DECR TPS | 평균 응답시간 변화 |
|---|---|---|---|
| 1,000건 | 499.8/sec | 593.8/sec (+19%) | 비슷 (저부하에선 Redis 왕복 비용 상쇄) |
| 3,000건 | 707.5/sec | 890.5/sec (+26%) | 2,234ms → 1,824ms |
| 6,000건 | 772.6/sec | 844.4/sec (+9%) | 3,922ms → 3,315ms |
| **10,000건** | 622.4/sec | **882.9/sec (+42%)** | **7,866ms → 5,137ms (-35%)** |

고부하일수록 Redis 1차 필터의 효과가 커지는 것을 확인했습니다. 반대로 **저부하에서는 Redis 네트워크 왕복이 추가 비용**이라
이점이 작다는 한계도 함께 기록했습니다 — 이 구조는 "트래픽이 몰리는 선착순 오픈"이라는 시나리오에 맞춘 선택입니다.

### 4-2. Redis ↔ DB 재고 정합성 — 트랜잭션 경계 트러블슈팅

**AS-IS**

주문 취소 시 DB 재고 복구와 Redis 재고 복구를 한 트랜잭션 안에서 함께 수행했습니다.
그런데 **Redis는 DB 트랜잭션에 참여하지 않기 때문에**, Redis 복구 후 DB 커밋이 롤백되면
Redis 재고만 늘어나는 정합성 깨짐(유령 재고)이 발생할 수 있었습니다.

또 하나의 트러블슈팅: 주문 생성 로직에서 Redis 차감(트랜잭션 밖)과 DB 차감(트랜잭션 안)을 같은 서비스의 내부 메서드 호출로 분리했더니,
**Spring AOP 프록시를 거치지 않는 자기 호출(self-invocation)이라 `@Transactional`이 적용되지 않는 문제**를 겪었습니다.

**TO-BE**

- 주문 취소의 Redis 재고 복구를 `@TransactionalEventListener(phase = AFTER_COMMIT)`로 분리 —
  **DB 커밋이 확정된 후에만** Redis를 복구하므로 롤백 시 유령 재고가 생기지 않습니다.
  Redis 복구 실패는 예외를 전파하지 않고 로그로 남겨, 이미 커밋된 주문 취소가 Redis 장애에 영향받지 않도록 했습니다.
- 주문 생성의 DB 트랜잭션 구간을 `OrderCreateService`라는 **별도 빈으로 분리**해 프록시를 정상 경유하도록 수정하고,
  롤백 시 Redis 복구가 동작하는지 **롤백 테스트를 추가**해 검증했습니다.

**배운 점**: "트랜잭션이 걸려 있다"와 "실제로 동작한다"는 다르며, 트랜잭션에 참여하지 않는 외부 저장소(Redis)는
**커밋 시점을 기준으로** 후속 처리를 설계해야 한다는 것을 체득했습니다.

### 4-3. 결제 금액 검증 및 중복 결제/취소 방지

**AS-IS**

결제 API가 클라이언트가 보낸 금액을 그대로 신뢰했습니다. 클라이언트 조작으로 **1원 결제 후 주문 완료**가 가능한 구조였고,
같은 주문에 결제와 취소 요청이 동시에 들어오면 상태가 꼬일 수 있었습니다.
검증 전에 Payment를 먼저 저장하는 순서 문제로, 검증 실패 시에도 결제 레코드가 남는 버그도 있었습니다.

**TO-BE**

- 결제 요청 금액과 서버가 계산한 `order.totalPrice`를 **서버 측에서 비교 검증**하고, 불일치 시 결제를 거부 (`PAYMENT_AMOUNT_MISMATCH`)
- **검증 통과 후에만** Payment를 저장하도록 순서 변경
- 결제·취소 모두 `Order`를 **비관적 락으로 조회**해 같은 주문에 대한 동시 결제/동시 취소를 단일 트랜잭션으로 직렬화
- 주문 상태 전이를 엔티티 메서드(`markAsPaid`, `cancel`)에 캡슐화해 허용되지 않는 상태 전이를 도메인에서 차단
- 취소 시 결제 상태(환불), 배송 상태까지 함께 전이시키고, 모든 상태 변경을 `OrderStatusHistory`로 기록해 **추적 가능성** 확보

### 4-4. JWT 인증과 Refresh Token 관리 — TTL 불일치 트러블슈팅

**AS-IS**

JWT 도입 초기, Refresh Token의 **JWT 자체 만료 시간과 Redis TTL이 달라**
"토큰은 유효한데 Redis에는 없는" / "Redis에는 있는데 토큰은 만료된" 불일치 구간이 존재했습니다.
또 Access Token이 서버 측에서 무효화되지 않아 로그아웃 후에도 사용 가능한 문제가 있었습니다.

**TO-BE**

- Refresh Token의 JWT 만료 시간과 Redis TTL을 단일 설정(`JwtProperties`)에서 관리해 **항상 일치**하도록 수정
- 로그아웃 시 Redis의 Refresh Token을 삭제해 재발급 차단
- 로그아웃한 Access Token은 **남은 유효시간만큼만 TTL을 걸어 Redis 블랙리스트에 등록** —
  모든 요청에서 JwtFilter가 블랙리스트를 확인해 즉시 무효화하면서, 만료된 토큰은 자동 삭제되어 Redis 부담 최소화
- Refresh Token 저장소 + Access Token 블랙리스트로 **stateless JWT의 장점(서버 확장성)과 강제 만료 가능성(보안)** 사이의 트레이드오프를 절충

### 4-5. 운영 환경 구성

**AS-IS**: 로컬에서만 동작하는 애플리케이션으로, 배포 시 환경마다 설정이 달라 재현이 어려웠습니다.

**TO-BE**: Dockerfile + Docker Compose로 앱·MySQL·Redis를 한 번에 기동하는 구성을 만들고 AWS EC2에 배포했습니다.
환경별 설정은 환경 변수로 분리해 로컬과 운영의 차이를 코드가 아닌 설정으로 흡수했습니다.

## 5. 이 프로젝트로 보여드리고 싶은 것

1. **정합성에 대한 집요함** — "동시성 테스트가 통과한다"에서 멈추지 않고, 락 전략 3가지를 직접 비교하고 트레이드오프를 근거로 선택했습니다.
2. **장애·엣지 케이스 중심 사고** — 롤백 시 유령 재고, 자기 호출 트랜잭션 미적용, TTL 불일치처럼 정상 흐름 밖의 문제를 찾아 고쳤습니다.
3. **검증하는 습관** — 동시성 테스트, 롤백 테스트를 작성해 수정이 실제로 동작함을 코드로 증명했습니다.
